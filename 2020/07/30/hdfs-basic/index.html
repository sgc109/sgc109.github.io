HDFS 란?

HDFS(Hadoop Distributed File System)은 이름 그대로 하둡(Hadoop) 의 분산 파일 시스템이다.

본 글에서는 HDFS 에 대해 알아본다.

HDFS 란?

B-Tree 기반의 RDBMS 는 쓸 때마다 배열을 재조정해야해서 어쩌구 저쩌구

하둡은 한 번 쓰고 여러번 읽을 때 유리하고, RDBMS 는 임의의 위치에 자주 수정할 때 어쩌구

빠른 응답시간이 요구되는 시나리오에는 적합하지 않다. 보통 수 분이 걸린다.

HDFS 가 왜 필요한가?

HDFS 는 기존의 RDBMS 와 비교하여 다음과 같은 장점이 있다.

확장성

HDFS 는 scale-out 하기 좋다.

HDFS 에서는 각 파일을 블록이라는 단위로 쪼개어 분산 클러스터에 나눠 저장한다.

그렇기 때문에 하나의 노드에 저장될 수 없는 아주 큰 파일이라도 여러 블록으로 쪼개지고, 여러 노드를 사용하여 저장할 수 있다.

또한, 클러스터를 이루는 노드로 고가의 장비를 사용하지 않아도 훌륭한 성능을 보여준다.

안전성

하나의 파일을 작게 쪼개어 여러 노드에 분산하여 저장하기 때문에 특정 노드의 장애로 인한 위험을 분산시킬 수 있다

또한, HDFS 는 각 블록을 저장할 때 보통 여러 개의 복사본(replication)을 만들어 여러 노드에 골고루 분산하여 저장한다. (기본적으로 3개의 복사본을 만들며, 변경 가능하다)

그렇기 때문에 데이터를 안전하게 저장할 수 있으며, 만에 하나 어떤 노드가 고장나서 데이터를 잃어 버려도 다른 노드에 저장된 복사본을 사용하면 문제 없이 동작할 수 있다.

속도

분산 저장된 파일들을 읽거나 쓸 때 발생하는 부하를 여러 노드로 분산시켜 동시에 다수의 컴퓨팅 리소스를 동시에 활용할 수 있어 대용량 데이터 처리 성능을 향상시킨다.

맵리듀스(MapReduce)와 같은 분산 애플리케이션과 함께 사용하면 장점이 극대화 된다.

사실 RDBMS 에서도 단일 노드의 한계를 극복하기 위해 복제(replication)와 분할(partitioning) 기능을 제공하는 경우가 많다.

하지만 이러한 기능은 나중에 고려된 것이며, 애초에 RDBMS 는 대규모 확장성과 분산을 염두에 두고 개발된 것이 아니기 때문에

확장함에 따라 기존 RDBMS 의 장점을 점점 잃게 되며, 설치, 유지보수, 그리고 비용 면에서 그리 효율적이지 못하다.

HDFS 는 어떻게 동작하나?

블록 하나는 기본적으로 512MB. 더 적은 데이터라도 기본 단위 만큼의 공간을 차지하는 디스크 파일 시스템과는 달리, 더 적은 데이터는 해당 크기의 블록이 만들어짐.

NameNode 와 DataNode

우선 HDFS 에는 둘은 마스터-워커 패턴으로 동작하는 2개의 장기 실행 데몬인 네임 노드(NameNode) 와 데이터 노드(DataNode) 가 있다.

HDFS 에서 파일은 블록 단위로 쪼개져 분산 저장되기 때문에 각 파일을 이루는 블록들이 어느 노드에 저장되었는지 알아야 나중에 파일을 읽을 때 다시 조립할 수 있을 것이다.

블록이 저장된 노드에 대한 정보는 시스템이 시작할 때 모든 데이터 노드로 부터 정보를 받아 재구성하기 때문에 디스크에 저장하지는 않는다.

또, HDFS 에서 각 파일이 실제로는 여러개의 블록으로 쪼개져 분산 저장되지만, HDFS 는 논리적인 디렉터리 구조를 사용한다.

이를 '네임 스페이스'라고 하는데, 이 때 각각의 파일이 어느 디렉터리에 있는지 알아야 한다. 네임 노드는 이 네임 스페이스와 파일시스템 트리에 속한 모든 파일과 디렉터리의 메타데이터 정보를

에디트 로그와 네임스페이스 이미지라는 형태로 로컬 디스크에 저장한다.

그래서 예를 들어 HDFS 클라이언트가 디렉터리 구조의 특정 위치에 파일을 쓸 때 네임노드는 해당 위치에 이미 파일이 존재하는지 확인하는 일과 새로운 파일이 생성되었다는 메타데이터를 업데이트하는 일을 하게된다.

또한, 클라이언트가 파일을 읽을 때는 해당 파일을 구성하는 블록이 위치한 노드를 알려주기도 한다.

한편, 데이터 노드는 실제로 블록이 저장되는 노드이다.

Secondary, Active, Backup NameNode

노드 간의 거리

클러스터 내 노드간의 거리는 대역폭을 기준으로 한다. 모든 노드 쌍 사이의 거리를 구하기는 어렵기 때문에 하둡에서는 네트워크 토폴로지를 트리 구조로 단순화하여 거리를 측정한다.

데이터 센터, 랙, 노드 순으로 계층 구조로 추상화 한다. 서로 다른 두 노드의 거리는 각 노드에서 최소 공통 조상 까지의 거리의 합이다.

예를 들어, 같은 노드는 당연히 거리가 0 이고(e.g. 같은 노드 내 두 블록의 경우), 같은 랙에 속한 두 노드는 거리가 2, 같은 데이터 센터에 있지만 다른 랙에 속한 두 노드는 거리가 4, 서로 다른 데이터 센터에 있는 두 노드는 거리가 6 인 식이다.

[그림 삽입]

참고로, 서로 다른 데이터 센터를 하나의 클러스터로 운영하는 것은 권장되지 않으며, 기본적으로는 단일 데이터 센터, 단일 랙에서 동작하도록 설정되어 있다.

네트워크 대역폭은 한정된 자원이기 때문에 전반적인 속도를 위해 최대한 아껴 쓰는 것이 중요하다. 그래서 하둡에서는 데이터 지역성(Data Locality) 이라는 것을 활용하여 네트워크 대역폭은 최소화하고, 처리 속도는 최대화하는데,

데이터 블록을 읽거나 쓸 때라 던지, 태스크를 실행할 노드를 배정할 때 가능한 거리가 가까운 노드를 선택한다.
